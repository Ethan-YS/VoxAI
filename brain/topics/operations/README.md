# operations/

> **这里放什么**：运维、流程、打包、部署、操作手册。
>
> **判断标准**：问自己——"这个文档回答的问题是**怎么操作它 / 每次要做什么**吗？"

---

## 典型内容

- **发版流程**：打包命令、上架步骤、RELEASE_CHECKLIST
- **打包 / 构建**：跨平台打包、签名、Notarization
- **部署日志**：某次部署的步骤和遇到的问题
- **运维操作**：重置用户数据、管理员任务、支持流程
- **变更历史**：CHANGELOG（按版本组织）
- **外部协作**：和其他 repo / 团队的对接说明

## 不属于这里的

- ❌ 系统设计 / 架构 → `systems/`
- ❌ 未来计划 / 路线图 → `planning/`
- ❌ bug 追踪 → `feedback/`
- ❌ **PRODUCT_PLAN 放项目根**，不进 operations/（PRODUCT_PLAN 是产品层级的"做什么"，operations 是操作层级的"怎么做"）

## 文件命名建议

- 大写字母 + 下划线，如 `RELEASE_CHECKLIST.md` / `PAYMENT_DEPLOY_LOG.md`
- 操作类动词开头或场景描述，让人一眼知道"什么时候翻这份"
- `CHANGELOG.md` 是惯例，保持下划线或不用下划线都行，看项目风格

## 维护纪律

- **RELEASE_CHECKLIST**：每次发版走完对照一遍；发现缺漏追加
- **CHANGELOG**：每次发版同步更新，不要等到"以后补"
- **DEPLOY_LOG**：每次重大部署当场记录，写具体步骤 + 遇到的问题
- 操作文档的"权威性"要靠**最后一次真的走通**来验证——久了没用过的流程默认已失效
