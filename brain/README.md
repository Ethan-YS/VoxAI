# brain/

这个文件夹装的是**关于这个项目本身**的所有协作文档——给未来的 Sage / 新会话 / 接手的人接续用。

## 接续层（每次开窗口要读）

直接放在 `brain/` 下的 5 份核心：

| 文件 | 回答什么 | 时间尺度 |
|---|---|---|
| `PROJECT.md` | 项目是什么？为什么存在？**不做什么**？ | 几乎不变 |
| `MAP.md` | 项目长什么样？想找什么信息去哪？ | 随结构演进 |
| `STATUS.md` | 现在停在哪？下一步？什么卡住了？ | 瞬时，随时覆写 |
| `DECISIONS.md` | 为什么做了 X？为什么没做 Y？ | 只追加 |
| `HANDOFF.md` | 上一个 Sage 切窗口前留给我的最新交接 | 切窗口时生成 |

`handoffs/` — 历史 HANDOFF 归档，按 `YYYY-MM-DD-HHMM.md` 命名。

## 专题层（按需触发读）

`topics/` 下的 4 分类——每份业务/技术文档按"面向什么问题"放进对应分类：

- `systems/` — "这个东西**是怎么设计的**"
- `operations/` — "**怎么操作**它 / 每次要做什么"
- `planning/` — "**将来要做**什么 / 怎么规划"
- `feedback/` — "**用户 / 现实**告诉我们什么"

进哪份的判断依据见 `topics/README.md`。

## 新会话怎么用

1. 先读 `MAP.md` — 了解项目结构 + 知道文档都在哪
2. 再读 `STATUS.md` — 知道现在进行到哪
3. 如果存在 `HANDOFF.md` — 读，知道上一个 Sage 留下的"新鲜信息"
4. 按需读 `PROJECT.md` / `DECISIONS.md` — 理解范围 / 追溯原因
5. 按 `MAP.md` 第五节的专题文档索引，进入 `topics/` 下的具体内容

## 维护纪律

- **接续层 5 份彼此不重叠**：任何信息都应该明确属于其中一份。混合会让系统退化。
- **PROJECT 不该频繁变**：若在频繁变，说明项目定位在漂移——这件事应该记进 DECISIONS。
- **STATUS 硬上限 80 行**：超了就沉淀到 MAP（结构变化）或 DECISIONS（决策原因），然后覆写。
- **DECISIONS 只追加**：历史条目永远不修改。即使决策被推翻，也用新条目推翻，保留旧条目。
- **HANDOFF 切窗口前覆写**：上一份归档到 `handoffs/`，主文件永远是当前最新。
