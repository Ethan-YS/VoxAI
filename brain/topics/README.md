# topics/

这个目录装的是项目的**业务/技术专题文档**——按"面向什么问题"分四类，不按业务模块分。

## 四分类与判断依据

| 子目录 | 放什么 | 判断标准（问自己） |
|---|---|---|
| `systems/` | 系统设计、架构、技术选型 | "这个文档回答的问题是**这个东西是怎么设计的**吗？" |
| `operations/` | 运维、流程、打包、部署 | "**怎么操作它 / 每次要做什么**吗？" |
| `planning/` | 路线图、定价策略、计划类 | "**将来要做什么 / 怎么规划**吗？" |
| `feedback/` | 用户反馈、bug 追踪、triage | "**用户 / 现实告诉我们什么**吗？" |

每个子目录有自己的 README，写明"放什么、不放什么、命名建议、维护纪律"。

## 为什么按"问题维度"而不按"业务模块"分

模块会增减、名字会变、会合并拆分；但"设计 / 操作 / 规划 / 反馈"这四个维度稳定得多。

一个业务模块（比如"支付系统"）的文档会散在四个子目录：
- 设计文档进 `systems/`
- 部署日志进 `operations/`
- 定价策略进 `planning/`
- 用户反馈进 `feedback/`

但每次要找的时候很清楚去哪找。

## 和接续层（brain/ 直接子文件）的区别

接续层 5 份核心（PROJECT/MAP/STATUS/DECISIONS/HANDOFF）是**每次开窗口都要读**的元信息。
专题层（这里）是**按需触发读**的具体内容——MAP.md 第五节会登记每份专题文档的"何时读"触发条件。

## 维护纪律

- 每份顶部写"这份文件回答：xxx"说明边界
- 新加文档要登记到 `brain/MAP.md` 第五节
- 删除/废弃文档要从 MAP 第五节同步移除
- 不维护"最后更新于"字段——协议保证文件最新；如果真的要查，用 `git log -1 --format=%ad <path>`
