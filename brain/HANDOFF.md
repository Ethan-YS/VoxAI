# HANDOFF — 跨窗口交接

> **这份文件回答**："上一个 Sage 切窗口前在做什么？还没沉淀进 STATUS 的新鲜信息有哪些？我从哪里接？"
>
> **特性**：
> - **瞬态**。Rebecca 说"切窗口"时由当前 Sage 写。
> - **下次切窗口前**，当前 Sage 先把现有 `HANDOFF.md` `git mv` 到 `handoffs/<它的最后修改时间戳>.md`，再写新的。
> - **极轻**。STATUS 已经覆写完整时，HANDOFF 可以非常短甚至几乎为空。
>
> **谁该读**：新窗口启动时的 Sage（如果文件存在）。

## 2026-05-06 上架后小结

STATUS 已经覆写完整反映"已上架"。这里只留几条**没法沉到 STATUS 的事实**：

1. **审核速度异常快**——提交（9:12）到通过 / 可分发是**同一天**。预期 1-3 天 → 实际 < 12 小时。这个数据点未来 v1.x 提交时可作为基线参考，但单点样本，不要 over-fit。

2. **新窗口的 Sage 接续时如果遇到"用户问怎么再发版本"**：路径是 **新建 1.x 版本（不在原版本下加 build）**。ASC sidebar 的"+ 新建版本"按钮——填新 What's New + 新 build（1.x.0）+ 重走出口合规（如果加了网络 / TLS 用法）+ 重新提交。这块流程已经熟，不再重头摸。

3. **如果用户报"App Store 搜不到"** 而 ASC 显示已可分发：先试直链 `https://apps.apple.com/cn/app/id6766570591` 确认产品页可达。可达 = 已上架，只是搜索索引延迟（0-24h）。不可达 = 区域限制 / 还在 propagate。

没有未解决的 bug，没有 in-flight 改动需要交接。
