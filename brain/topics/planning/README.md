# planning/

> **这里放什么**：路线图、定价策略、计划类文档。
>
> **判断标准**：问自己——"这个文档回答的问题是**将来要做什么 / 怎么规划**吗？"

---

## 典型内容

- **定价策略**：分层设计、Stripe 配置、买断 vs 订阅的权衡
- **路线图 / 阶段计划**：按阶段 / Sprint 的功能规划
- **重构实施计划**：跨多次 commit 的大改造方案（完成后标注为"历史文档"）
- **产品策略细节**：PRODUCT_PLAN 之外的深度规划文档

## 不属于这里的

- ❌ PRODUCT_PLAN 本身 → **项目根**（和 README 并列，是项目门面级文档）
- ❌ 当前瞬时状态 → `brain/STATUS.md`（STATUS 是"现在在做"，planning 是"将来做什么"）
- ❌ 已执行完的部署 → `operations/`
- ❌ 历史决策原因 → `brain/DECISIONS.md`

## 和 `brain/STATUS.md` 的区别

| 维度 | `brain/STATUS.md` | `planning/*.md` |
|---|---|---|
| 时间尺度 | 现在 / 这几天 | 未来几周 / 几个月 |
| 更新频率 | 每次会话 | 偶尔 |
| 内容 | 具体到"下一步改哪个文件" | 抽象到"Q2 要把支付闭环做完" |
| 长度 | 硬上限 ~80 行 | 没有硬上限 |

**简单判断**：会话结束前要覆写的内容放 STATUS，跨会话稳定的计划放 planning/。

## 和 `DECISIONS.md` 的区别

- `planning/` = **还没做**的计划、**准备做**的方向
- `brain/DECISIONS.md` = **已经决定**的事 + 为什么这么决定 + 为什么没选别的

计划书里的某个方向一旦被**拍板决定**，相关决策条目应该同步进 DECISIONS。

## 维护纪律

- 执行完的计划 → 顶部打"**历史文档**"标签，保留但不再改
- 计划调整 → 不直接覆盖，应该在 DECISIONS 追加"方向调整"决策条目
- 定价之类的**生效策略**（不是规划，而是当前正在执行的定价模型）可以放这里，但要标"当前生效"
